Seed orders + auto-progression

Each of the 5 users gets 2–3 pre-seeded orders. Timestamps are set as datetime.now() - timedelta(...) so they stay accurate regardless of when the server starts. A third order is optional per user.

Order slotplaced_at offsetStarting statusFin scenario covered
ORD-1 (all users) 3 days ago delivered Return / replacement — B1, B2, B4
ORD-2 (all users) 20 min ago dispatched Reschedule delivery — A7, A8
ORD-3 (optional, some users) 5 min ago processing Cancel / address update — A3, A6

Status is never stored — it's derived on every GET /api/orders/{id} call from elapsed time since placed_at. The only exception is cancelled, which is a stored flag set by A3 or admin override.

Checkout flow + order structure

1
Browse shop.html
Click "Add to Bag" → calls E2
2
Review cart.html
Items, quantities, subtotal via E1. Update via E3. Remove via E4
3
Click "Place Order"
E5 called. Default address used. No payment flow
4
Order created
ORD-XXXXX created. Cart cleared. placed_at = now
5
Redirect + Admin update
Redirect to orders.html. Admin panel picks up new order within 5s
One order per checkout — all cart items become line items in a single ORD-XXXXX. Big and small items can be mixed. Delivery timeline is determined by the largest item: if any large furniture item (sofa, bed, dining table) is in the cart → shipping_method = large_item, estimated_delivery = placed_at + 10 days. All other carts → shipping_method = standard, estimated_delivery = placed_at + 5 days.
Order fieldValue / logic
order_idAuto-generated — ORD-10001 onwards, increments
customer_idFrom request body
itemsArray of {product_id, product_name, qty, unit_price, line_total}
price_totalSum of all line totals in ₹
placed_atdatetime.now() at checkout time
shipping_methodlarge_item or standard — derived from cart contents
estimated_deliveryplaced_at + 10 days (large_item) or + 5 days (standard)
statusDerived — never stored. Computed from elapsed minutes on every GET
cancelledBoolean flag — the only stored status. Set by A3 or admin Z1
delivery_addressCustomer's default address at time of checkout

Reset Demo — what Z3 does

Wiped on reset
STATUS_OVERRIDES — cleared. All orders return to auto-progression
DYNAMIC_RETURNS — cleared. Returns created during demo removed
CARTS — cleared. All carts emptied
Checkout orders — any ORD-XXXXX created during the demo session removed
Preserved on reset
CUSTOMERS — hardcoded, never touched
PRODUCTS — hardcoded, never touched
PAYMENT_METHODS — hardcoded, never touched
Seed orders (ORD-10041 etc.) — always restored to original state
Seed returns (RET-2201/02/03) — always preserved
Reset = backend returns to exactly how it looked 1 second after the server booted. Safe to run between any two demo sessions without restarting Railway.
